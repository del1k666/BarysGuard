## АННОТАЦИЯ

Настоящая дипломная работа посвящена разработке программного комплекса для автоматизации поиска индикаторов компрометации и оперативного реагирования на инциденты информационной безопасности. Актуальность исследования обусловлена резким ростом числа кибератак: согласно отчёту Kaspersky Security Bulletin 2023, число попыток вредоносного заражения в корпоративном секторе увеличилось на 41% по сравнению с предыдущим годом. Существующие коммерческие SIEM-системы и платформы реагирования зачастую требуют значительных финансовых вложений и высокой квалификации персонала, что делает их недоступными для организаций среднего и малого масштаба.

В работе разработано десктопное приложение BarysGuard на языке Python с использованием фреймворка PyQt6. Система обеспечивает автоматический сбор индикаторов компрометации с локальных и удалённых узлов, статический анализ файлов посредством правил YARA, проверку хэш-сумм через VirusTotal API, сетевую разведку с применением AbuseIPDB, а также оперативную сетевую изоляцию скомпрометированных хостов через механизмы Windows Firewall. Удалённое управление реализовано через лёгкого агента на базе Flask с TLS-шифрованием.

Практическая ценность работы заключается в создании доступного инструмента для аналитиков безопасности и системных администраторов, позволяющего значительно сократить среднее время обнаружения угрозы (MTTD) и время реагирования (MTTR) без необходимости развёртывания сложной инфраструктуры.

**Ключевые слова:** индикаторы компрометации, реагирование на инциденты, YARA, автоматизация, кибербезопасность, threat intelligence, Windows Firewall, удалённый агент.

## ANNOTATION

This diploma thesis addresses the development of a software system for automated detection of indicators of compromise (IoC) and rapid incident response in information security. The relevance of the research stems from the sharp increase in cyberattack frequency: according to the Kaspersky Security Bulletin 2023, the number of malicious infection attempts in the corporate sector increased by 41% compared to the previous year. Existing commercial SIEM platforms and incident response tools often require significant financial investment and high personnel expertise, making them inaccessible to small and medium-sized organisations.

The study presents a desktop application, BarysGuard, developed in Python using the PyQt6 framework. The system provides automated collection of indicators of compromise from local and remote network nodes, static file analysis via YARA rules, hash verification through the VirusTotal API, network intelligence using AbuseIPDB, and rapid network isolation of compromised hosts through Windows Firewall mechanisms. Remote management is implemented via a lightweight Flask-based agent with TLS encryption.

The practical contribution of the work is the creation of an accessible tool for security analysts and system administrators that significantly reduces mean time to detect (MTTD) and mean time to respond (MTTR) without requiring deployment of complex infrastructure.

**Keywords:** indicators of compromise, incident response, YARA, automation, cybersecurity, threat intelligence, Windows Firewall, remote agent.

## ВВЕДЕНИЕ

Современная цифровая среда характеризуется постоянно возрастающей сложностью киберугроз и скоростью их распространения. Инциденты информационной безопасности — от целевых атак с применением программ-вымогателей до многоэтапных компрометаций корпоративных сетей — перестали быть редкостью и превратились в устойчивую операционную реальность для большинства организаций. Согласно данным Mandiant M-Trends 2024, медианное время нахождения злоумышленника внутри инфраструктуры жертвы до обнаружения атаки составляет 10 дней, а в ряде случаев вторжение остаётся незамеченным на протяжении нескольких месяцев (Mandiant, 2024). Каждый час, в течение которого угроза остаётся необнаруженной, увеличивает потенциальный ущерб от инцидента.

Ключевым инструментом своевременного обнаружения атак служат индикаторы компрометации (IoC) — наблюдаемые артефакты, свидетельствующие о факте нарушения безопасности или активной вредоносной деятельности в системе. К ним относятся хэш-суммы подозрительных файлов, IP-адреса командных серверов, доменные имена, сигнатуры в памяти процессов, аномальные записи реестра автозапуска и многое другое. Систематический и автоматизированный сбор подобных артефактов, их корреляция с известными угрозами и последующее реагирование — именно эти задачи лежат в основе современной оперативной кибербезопасности.

Вместе с тем практика показывает, что значительная часть организаций — особенно малые и средние предприятия, государственные учреждения с ограниченным бюджетом на ИБ — не имеют доступа к полноценным корпоративным решениям класса SIEM и SOAR. Согласно отчёту ENISA Threat Landscape 2023, 62% инцидентов в организациях среднего сегмента обнаруживаются постфактум, уже после реализации угрозы, поскольку инструменты мониторинга либо отсутствуют, либо настроены неэффективно (ENISA, 2023). Это формирует очевидный технологический разрыв, устранение которого представляет собой актуальную практическую задачу.

**Актуальность** настоящей работы определяется необходимостью создания доступного, функционально полного инструмента автоматизации поиска IoC и реагирования на инциденты, ориентированного на специалистов по безопасности без привязки к дорогостоящей коммерческой инфраструктуре.

**Цель работы** — разработка программного комплекса BarysGuard, обеспечивающего автоматизированный поиск индикаторов компрометации на локальных и удалённых узлах сети Windows, интеграцию с внешними разведывательными сервисами и механизмы оперативного реагирования, включая сетевую изоляцию скомпрометированных хостов.

Для достижения поставленной цели были определены следующие **задачи**:

1. Провести анализ теоретических основ и современных подходов к обнаружению IoC и реагированию на инциденты.

## ГЛАВА 1. ТЕОРЕТИЧЕСКИЕ ОСНОВЫ АВТОМАТИЗАЦИИ ОБНАРУЖЕНИЯ ИНДИКАТОРОВ КОМПРОМЕТАЦИИ

### 1.1 Понятие и классификация индикаторов компрометации

Термин «индикатор компрометации» (Indicator of Compromise, IoC) прочно вошёл в профессиональный лексикон специалистов по информационной безопасности с начала 2010-х годов, когда реактивная модель защиты — ожидание инцидента и его устранение постфактум — стала очевидно недостаточной. Концептуальную основу для систематической работы с IoC заложили исследователи компании Mandiant, предложившие в 2013 году формальную спецификацию OpenIOC как открытый стандарт описания признаков компрометации в машиночитаемом формате (Mandiant, 2024).

В общем определении IoC представляет собой наблюдаемый цифровой артефакт или поведенческий признак, с высокой степенью достоверности указывающий на факт несанкционированного доступа, активной вредоносной активности или подготовки к атаке в информационной системе (Luttgens et al., 2014). Принципиальное отличие IoC от общих признаков аномалии — их привязанность к конкретным угрозам: за каждым индикатором стоит либо известная вредоносная программа, либо установленная группировка, либо воспроизводимый паттерн атаки.

С точки зрения классификации IoC принято делить по нескольким основаниям. По **типу артефакта** выделяют:

- *Файловые индикаторы* — хэш-суммы вредоносных файлов (MD5, SHA-1, SHA-256), их имена, размеры и характерные строки в теле. Хэш-суммы являются наиболее точными индикаторами: они однозначно идентифицируют конкретный экземпляр вредоносного ПО, хотя злоумышленники научились обходить их путём незначительной модификации файла (Ligh et al., 2014).
- *Сетевые индикаторы* — IP-адреса командно-контрольных серверов (C2), подозрительные доменные имена и URL, характерные паттерны сетевого трафика. По данным VirusTotal Malware Trends Report 2022, более 70% вредоносного ПО в той или иной мере использует сетевые коммуникации для получения команд или эксфильтрации данных (VirusTotal, 2023).
- *Системные индикаторы* — нестандартные записи в ключах реестра автозапуска, аномальные запущенные процессы, нетипичные системные службы, подозрительные задачи планировщика.
- *Поведенческие индикаторы* — паттерны действий, характерные для конкретных TTPs (Tactics, Techniques and Procedures): инъекция кода в легитимные процессы, обход UAC, горизонтальное перемещение по сети, дамп учётных данных.

По **уровню достоверности** (согласно пирамиде боли Дэвида Бьянко) IoC ранжируются от легко изменяемых злоумышленником хэшей и IP-адресов до практически неизменяемых TTPs, что определяет долгосрочную ценность каждого типа индикатора для обнаружения угроз (Hutchins et al., 2011).

По **источнику** IoC делятся на внешние (получаемые из разведки угроз — threat intelligence feeds, отраслевых отчётов, ISAC) и внутренние (выявляемые в ходе мониторинга собственной инфраструктуры). Комбинирование обоих источников существенно повышает полноту и точность обнаружения.

Для формализованного описания IoC разработан ряд стандартов: OpenIOC (Mandiant), STIX/TAXII (OASIS), MISP (Malware Information Sharing Platform). Наиболее широкое практическое применение получила матрица MITRE ATT&CK, систематизирующая TTPs реальных групп угроз в виде тактик и техник с привязкой к конкретным фазам атаки (Strom et al., 2018). На сегодняшний день матрица ATT&CK содержит более 400 задокументированных техник для платформы Windows и является де-факто стандартом структурирования знаний об угрозах в отрасли.

Практическая работа с IoC предполагает три ключевых этапа: *сбор* артефактов из системы, *обогащение* путём сопоставления с известными угрозами и внешними базами данных, и *действие* — принятие решений об изоляции, блокировке или дальнейшем расследовании. Именно автоматизация этой цепочки составляет ядро разрабатываемой системы.

### 1.2 Жизненный цикл реагирования на инциденты информационной безопасности

Реагирование на инциденты — это структурированный процесс, позволяющий организации обнаруживать угрозы, сдерживать их распространение, устранять последствия и восстанавливать нормальную работу. Основополагающим нормативным документом в этой области служит руководство NIST SP 800-61 Rev. 2, определяющее жизненный цикл реагирования из четырёх фаз: подготовка, обнаружение и анализ, сдерживание/устранение/восстановление, постинцидентная деятельность (Cichonski et al., 2012).

**Фаза подготовки** включает формирование команды реагирования (CSIRT), разработку регламентов и процедур, создание инструментальной базы мониторинга и настройку систем сбора событий. По данным SANS Institute, организации, вложившие время в создание детальных playbook-сценариев реагирования, сокращают MTTR в среднем на 47% по сравнению с теми, кто действует в режиме импровизации (Bromiley, 2016).

**Фаза обнаружения и анализа** — наиболее критическая с точки зрения скорости. На этом этапе происходит выявление аномалий в потоке событий, корреляция признаков с известными угрозами, приоритизация инцидентов. Именно здесь автоматизация приносит наибольший эффект: ручная обработка журналов событий на нескольких десятках хостов занимает часы, тогда как автоматизированный сбор IoC позволяет получить первичную картину за считанные минуты. Согласно данным Mandiant, медианное MTTD для организаций с автоматизированными средствами обнаружения составило 7 дней в 2023 году против 21 дня для тех, кто полагается исключительно на ручной анализ (Mandiant, 2024).

**Фаза сдерживания** предполагает немедленные действия по ограничению распространения угрозы. Ключевым техническим инструментом здесь является сетевая изоляция скомпрометированного хоста — отключение его от остальной инфраструктуры при сохранении канала управления для аналитика. Правильная реализация изоляции позволяет предотвратить горизонтальное перемещение злоумышленника и защитить критически важные системы от заражения.

**Фаза восстановления** включает удаление вредоносных артефактов, восстановление из резервных копий и возврат сервисов в рабочее состояние. После завершения инцидента проводится детальный анализ произошедшего (Lessons Learned), результаты которого обновляют базу знаний команды и совершенствуют процедуры подготовки к следующим инцидентам.

Помимо модели NIST, в отрасли используется концепция «цепочки заражения» (Cyber Kill Chain), предложенная исследователями Lockheed Martin (Hutchins et al., 2011). Согласно этой модели, большинство целевых атак проходят через последовательные стадии: разведка, вооружение, доставка полезной нагрузки, эксплуатация, установка, создание C2-канала и реализация целей. Обнаружение IoC на любой из этих стадий — особенно ранних — принципиально важно для прерывания атаки до её завершения. Именно это концептуальное основание определяет необходимость мониторинга нескольких слоёв артефактов одновременно: файловых, сетевых и поведенческих.

Автоматизация реагирования (Security Orchestration, Automation and Response, SOAR) стала одним из главных трендов последнего десятилетия. Опрос SANS 2016 года зафиксировал, что 68% организаций в той или иной мере автоматизировали хотя бы один аспект реагирования, однако полная автоматизация оставалась редкостью (Bromiley, 2016). К 2024 году ситуация изменилась: автоматизация стала стандартом для крупных организаций, хотя МСП по-прежнему испытывают дефицит доступных инструментов.

### 1.3 Современные методы автоматизированного обнаружения угроз

Современный арсенал методов автоматизированного обнаружения угроз включает несколько принципиально различных подходов, каждый из которых обладает собственными преимуществами и ограничениями.

**Сигнатурный анализ** — наиболее зрелый метод, применяемый со времён первых антивирусных систем. Он основан на сопоставлении наблюдаемых артефактов с базой известных индикаторов угроз. YARA — язык описания вредоносных паттернов — является современным воплощением этого подхода, позволяя описывать как простые строковые сигнатуры, так и сложные комбинированные условия на основе бинарных паттернов, регулярных выражений и метаданных файлов (Pontiroli & Martinez, 2015). Главное преимущество сигнатурного анализа — высокая точность для известных угроз и отсутствие ложных срабатываний при качественно написанных правилах. Ограничение — неэффективность против новых, ранее не описанных угроз.

**Анализ на основе поведения** смещает фокус с конкретных артефактов на характер действий программы или пользователя. Обнаружение аномальных API-вызовов, нетипичных сетевых соединений или подозрительных операций с файловой системой позволяет идентифицировать угрозы даже при отсутствии известных сигнатур. Исследование Sommer и Paxson (2010) показало, что применение методов машинного обучения для поведенческого анализа сетевого трафика сопряжено с рядом специфических трудностей: высокой стоимостью ложных срабатываний в условиях реальной инфраструктуры и сложностью разграничения легитимных и вредоносных аномалий.

**Корреляция событий** — основной принцип работы SIEM-систем — позволяет выявлять угрозы, не очевидные по отдельным событиям, через их совокупность во времени и пространстве. Появление нового процесса, немедленно открывающего исходящее соединение на нестандартный порт, по отдельности может казаться нормальным, однако в сочетании с предшествующей загрузкой документа с макросом формирует устойчивый паттерн атаки.

**Threat Intelligence** — интеграция внешних данных об угрозах в процессы обнаружения. Службы вроде VirusTotal агрегируют результаты проверок десятков антивирусных движков и предоставляют API для программного обращения. AbuseIPDB хранит базу IP-адресов, ассоциированных с вредоносной активностью, с оценкой степени риска. Использование подобных сервисов существенно ускоряет первичную классификацию подозрительных артефактов: вместо локального анализа аналитик получает немедленный отклик от глобальной базы знаний (Abu et al., 2018).

**Форензика памяти** — извлечение и анализ содержимого оперативной памяти работающего процесса — позволяет обнаруживать угрозы, намеренно избегающие записи на диск (fileless malware). Именно в памяти хранятся расшифрованные полезные нагрузки, строки конфигурации C2-серверов и инжектированный шелл-код, которые невозможно обнаружить статическим сканированием файловой системы (Ligh et al., 2014).

Практическая реализация разрабатываемой системы объединяет три из перечисленных подходов: сигнатурный анализ через YARA, интеграцию с threat intelligence через VirusTotal и AbuseIPDB, а также базовую форензику памяти через сканирование исполняемых файлов запущенных процессов.

### 1.4 Анализ существующих решений и их сравнение

Рынок инструментов для работы с IoC и реагирования на инциденты достаточно широк, однако большинство зрелых решений ориентировано либо на крупные предприятия с развитым SOC, либо требует значительных усилий по внедрению и поддержке. Рассмотрим основные категории существующих решений.

**Коммерческие SIEM-системы.** Splunk Enterprise Security и IBM QRadar представляют собой полнофункциональные платформы, обеспечивающие сбор логов, корреляцию событий, визуализацию и автоматическое реагирование. Microsoft Sentinel — облачная SIEM следующего поколения, тесно интегрированная с экосистемой Azure. Их ключевые преимущества — масштабируемость, богатая экосистема интеграций и поддержка производителя. Основной недостаток — высокая стоимость: лицензия Splunk для организации с объёмом данных 10 ГБ/день исчисляется десятками тысяч долларов в год, что делает их недоступными для большинства небольших организаций.

**Решения с открытым исходным кодом.** Wazuh — open-source HIDS/SIEM-система, обеспечивающая мониторинг конечных точек, обнаружение вторжений и compliance-аудит. TheHive в связке с Cortex — платформа управления инцидентами с автоматизацией обогащения IoC. OSSIM (AlienVault) — комплексное SIEM-решение с открытым кодом. Эти инструменты предоставляют широкие возможности, но требуют значительных компетенций для установки, настройки и сопровождения: по экспертным оценкам, полноценное внедрение Wazuh в инфраструктуре из 50 хостов занимает от 2 до 4 недель.

## ГЛАВА 2. ПРОЕКТИРОВАНИЕ СИСТЕМЫ BARYSGUARD

### 2.1 Требования к системе и постановка задачи

Проектирование системы начиналось с формулировки требований, которые были разделены на функциональные и нефункциональные. Исходными данными послужили анализ существующих решений, приведённый в разделе 1.4, а также ряд практических сценариев использования, характерных для небольших SOC и системных администраторов.

**Функциональные требования:**

1. Система должна обеспечивать автоматический сбор IoC с локального хоста: перечень запущенных процессов с путями к исполняемым файлам, активные сетевые соединения, записи реестра автозапуска.
2. Система должна выполнять статический анализ файлов и каталогов с применением YARA-правил из встроенной базы (не менее 40 правил) и пользовательских правил.
3. Система должна поддерживать проверку хэш-сумм файлов (MD5, SHA-1, SHA-256) через VirusTotal API с отображением результатов обнаружения.
4. Система должна поддерживать проверку IP-адресов и доменных имён через AbuseIPDB API с отображением оценки угрозы.
5. Система должна обеспечивать удалённое управление несколькими Windows-хостами через защищённый HTTPS-агент.
6. Система должна поддерживать сканирование памяти запущенных процессов с применением YARA-правил.
7. Система должна реализовывать сетевую изоляцию удалённого хоста через Windows Firewall с сохранением канала управления.
8. Система должна предоставлять инструмент карантина для изоляции подозрительных файлов.
9. Система должна формировать отчёты об инцидентах в формате HTML.
10. Интерфейс системы должен поддерживать три языка: русский, английский, казахский.

**Нефункциональные требования:**

- *Производительность:* сбор IoC с локального хоста должен выполняться не более чем за 60 секунд при стандартной конфигурации системы.
- *Безопасность:* связь с удалёнными агентами должна осуществляться исключительно по TLS; каждый запрос должен аутентифицироваться токеном.
- *Надёжность:* отказ при работе с одним удалённым хостом не должен влиять на работу с другими.
- *Переносимость:* агентская часть должна распространяться как единый исполняемый файл без необходимости установки Python на целевом хосте.
- *Удобство использования:* установка и первоначальная настройка системы должны занимать не более 15 минут.

### 2.2 Архитектура приложения

Архитектура BarysGuard построена по принципу разделения ответственности между несколькими уровнями: пользовательский интерфейс, бизнес-логика рабочих потоков, ядро системы и удалённый агент.

[Рисунок 1 — Общая архитектура системы BarysGuard]

**Уровень представления (UI Layer)** реализован на PyQt6 и организован в виде набора вкладок главного окна. Каждая вкладка представляет отдельный функциональный модуль: Dashboard, Hash Lookup, IOC Collection, YARA Scanner, Network Intel, Report Builder, Memory Scan, Quarantine, Settings, Hosts. Все компоненты UI следуют единому стилю оформления, заданному через глобальную таблицу стилей Qt. Главное окно также реализует механизм интернационализации через файл переводов `i18n.py`, поддерживая переключение языка в реальном времени без перезапуска приложения.

**Уровень рабочих потоков (Workers Layer)** отвечает за выполнение длительных операций в фоновых потоках с целью сохранения отзывчивости UI. Каждый рабочий поток наследует `QThread` и взаимодействует с UI исключительно через сигналы Qt. Разработаны следующие воркеры: `IOCWorker`, `YARAWorker`, `HashWorker`, `NetWorker`, `HostWorker`, `ProcessWorker`.

**Уровень ядра (Core Layer)** содержит общую инфраструктуру: модуль конфигурации (`config.py`), клиент удалённого агента (`agent_client.py`), движок YARA (`yara_engine.py`), менеджер хостов (`hosts_config.py`), утилиты хэширования (`hash_utils.py`).

**Удалённый агент (Remote Agent)** — самостоятельный компонент, распространяемый как скомпилированный исполняемый файл `agent.exe`. Реализован на базе Flask с поддержкой HTTPS через самоподписанный сертификат. Агент запускается как служба Windows (SYSTEM-привилегии) и экспонирует REST API из 10 эндпоинтов для выполнения сканирований, сбора данных и управления изоляцией.

Выбор технологического стека определялся требованиями к нативной поддержке Windows, доступности библиотек для задач кибербезопасности и возможности компиляции в единый исполняемый файл.

Правила YARA применяются практически во всех крупных антивирусных лабораториях, включая Kaspersky, ESET, и представляют собой удобный декларативный язык для описания как простых строковых сигнатур, так и сложных бинарных паттернов (Pontiroli & Martinez, 2015).

### 2.4 Проектирование модулей системы

Система разделена на функционально независимые модули, взаимодействующие через чётко определённые интерфейсы. Такая организация кода снижает связность компонентов и упрощает изолированное тестирование.

**Модуль сбора IoC** (`workers/ioc_worker.py`) отвечает за сбор трёх категорий артефактов: процессы (через `psutil.process_iter`), сетевые соединения (`psutil.net_connections`) и записи автозапуска реестра (через `winreg`). Обнаружение подозрительных процессов основано на эвристике расположения: исполняемые файлы за пределами системных директорий (`C:\Windows`, `C:\Program Files`) помечаются как потенциально аномальные. Результаты сохраняются в структурированные CSV-файлы в каталоге результатов.

**Модуль YARA-сканирования** (`workers/yara_worker.py`, `core/yara_engine.py`) реализует сканирование файлов и каталогов с применением скомпилированного набора правил. Движок поддерживает два источника правил: встроенные правила из `constants.py` (40+ правил) и пользовательские правила, написанные в редакторе приложения.

**Модуль удалённого управления** включает клиент (`core/agent_client.py`) и сервер (`agent/agent.py`). Клиент реализует HTTP-запросы к REST API агента с аутентификацией через заголовок `X-Api-Token` и игнорированием верификации самоподписанного сертификата — приемлемая мера безопасности для изолированных корпоративных сетей.

**Модуль сетевой изоляции** (`agent/agent.py`, эндпоинты `/network/isolate` и `/network/restore`) реализует изоляцию через PowerShell-командлеты Windows Firewall. Подробнее этот модуль описан в разделе 3.4.

## ГЛАВА 3. РЕАЛИЗАЦИЯ СИСТЕМЫ

### 3.1 Модуль сбора индикаторов компрометации

Модуль сбора IoC является центральным компонентом системы и реализует автоматизированный процесс извлечения артефактов из операционной системы. Работа модуля организована как фоновый поток `IOCWorker`, наследующий `QThread`, что обеспечивает выполнение длительных операций без блокировки UI.

Сбор процессов выполняется через итерацию `psutil.process_iter` с извлечением атрибутов: PID, имя процесса, полный путь к исполняемому файлу, командная строка запуска, имя пользователя. Для каждого процесса выполняется проверка расположения: если путь не начинается с `C:\Windows` или `C:\Program Files`, процесс помечается тегом `[SUSPICIOUS]`. Подобная эвристика, несмотря на свою простоту, эффективна для первичной фильтрации — подавляющее большинство легитимных системных процессов расположено в стандартных директориях, тогда как вредоносное ПО часто размещается во временных каталогах, директории `AppData` или корне системного диска.

[Рисунок 3 — Вкладка IOC Collection: результаты сбора с подсветкой подозрительных процессов]

Сбор сетевых соединений производится через `psutil.net_connections(kind='inet')`, что позволяет получить полный список активных TCP и UDP-соединений с указанием локального и удалённого адреса, порта и статуса (ESTABLISHED, LISTEN, TIME_WAIT и т.д.). Наличие соединений с нестандартными внешними адресами, особенно в сочетании с подозрительным процессом, является значимым IoC — признаком возможной C2-коммуникации.

Анализ записей автозапуска осуществляется посредством обхода ключевых ветвей реестра Windows: `HKCU\Software\Microsoft\Windows\CurrentVersion\Run`, `HKLM\Software\Microsoft\Windows\CurrentVersion\Run`, а также `HKLM\SYSTEM\CurrentControlSet\Services`. Именно эти ключи являются наиболее распространёнными векторами закрепления вредоносного ПО в системе — через них обеспечивается автоматический запуск при каждой загрузке системы (Kaspersky Lab, 2024).

Все собранные данные сохраняются в CSV-файлы в пользовательском каталоге результатов. Статистика — суммарное число процессов, соединений, записей автозапуска и подозрительных объектов — отображается в виде информационных карточек на вкладке IOC и синхронизируется с лентой событий Dashboard через статический метод `DashboardTab.log_event`.

### 3.2 Подсистема YARA-сканирования

YARA-сканер является одним из наиболее технически сложных компонентов системы. Его реализация опирается на официальную библиотеку `yara-python` версии 4.5+ и поддерживает как файловое, так и процессно-ориентированное сканирование.

**База встроенных правил** охватывает 40+ категорий угроз, включая:
- Шифровальщики: LockBit (правила на основе строк конфигурации и паттернов шифрования RSA), BlackCat/ALPHV (детектирование Rust-специфичных метаданных и строк конфигурации JSON), WannaCry (сигнатуры Eternal Blue exploit и встроенных текстовых артефактов);
- RAT-программы: AsyncRAT, QuasarRAT, Remcos — с правилами на основе строк протокола управления и характерных имён классов .NET-сборок;
- Стилеры: RedLine Stealer, Vidar, AgentTesla — с детектированием через строки API Telegram, специфичных URL и характерных секций PE-файла;
- C2-фреймворки: Sliver (детектирование Go-специфичных строк импорта), Havoc C2, Chisel Tunnel, Cobalt Strike Beacon;
- Инструменты атаки: Mimikatz (сигнатуры LSASS-дампа и строки sekurlsa), AMSI-bypass техники, отключение Windows Defender;
- LOLBins-злоупотребления: нетипичное использование `certutil.exe`, `regsvr32.exe`, `mshta.exe` для загрузки полезных нагрузок.

Каждое правило содержит поле `meta` со значениями `description`, `author` и `severity` (critical/high/medium/low), что позволяет системе автоматически классифицировать находки по уровню опасности.

[Рисунок 4 — Вкладка YARA Scanner: результаты сканирования с цветовой маркировкой по severity]

Движок `yara_engine.py` компилирует все активные правила в единый объект `yara.Rules` при запуске сканирования, что значительно повышает производительность по сравнению с пошаговой компиляцией. Сканирование каталогов выполняется рекурсивно с поддержкой прерывания через флаг `_stop`.

**Редактор пользовательских правил** встроен непосредственно в интерфейс вкладок File Scan и Memory Scan: пользователь пишет правило в текстовом редакторе, нажимает «Добавить/Обновить» и правило немедленно появляется в списке доступных для сканирования. Имя правила извлекается автоматически через регулярное выражение из объявления `rule <Name>`.

### 3.3 Удалённое управление хостами и агент

Возможность централизованного управления несколькими хостами — ключевая архитектурная особенность BarysGuard, отличающая его от большинства аналогичных инструментов в данном сегменте.

**Архитектура агента.** `agent.exe` — это самодостаточное Flask-приложение, скомпилированное через PyInstaller в единый исполняемый файл размером около 15 МБ. При первом запуске агент генерирует самоподписанный RSA-2048 сертификат и 64-символьный токен аутентификации, сохраняя их в рабочей директории. Поддерживается установка как служба Windows (`--install`, `--start`) с использованием библиотеки `pywin32`, что обеспечивает автоматический запуск при загрузке и работу под учётной записью SYSTEM — необходимое условие для получения привилегий, достаточных для сетевой изоляции.

Агент экспонирует следующие REST-эндпоинты:

| Метод | Путь | Описание |
|---|---|---|
| GET | /ping | Проверка доступности, hostname, OS, версия |
| GET | /info | CPU/RAM/диск, залогиненные пользователи |
| POST | /scan/yara | YARA-сканирование удалённого пути |
| POST | /scan/ioc | Сбор процессов, соединений, автозапуска |
| POST | /scan/memory | Сканирование памяти процесса |
| POST | /scan/hashes | Хэши файлов в указанном пути |
| GET | /processes | Список запущенных процессов |
| GET | /network/status | Статус изоляции (isolated: true/false) |
| POST | /network/isolate | Применить сетевую изоляцию |
| POST | /network/restore | Снять изоляцию |

**Вкладка Hosts** предоставляет интерфейс для управления реестром хостов. Каждый хост представлен записью с полями: имя, IP-адрес, порт, токен аутентификации, статус последнего пинга и время последнего сканирования. Хосты хранятся в файле `hosts.json`. Для каждого хоста доступны пять вложенных вкладок: Status, File Scan, Memory Scan, Results, Isolation.

[Рисунок 5 — Вкладка Hosts: список хостов и подвкладки управления]

**Автоматизированное развёртывание агента** реализовано через протокол WinRM с использованием библиотеки `pywinrm`. Оператор вводит IP-адрес целевого хоста, имя и пароль администратора — система автоматически копирует `agent.exe`, устанавливает службу и получает токен. Эта возможность позволяет подключить новый хост к системе мониторинга за считанные минуты без физического доступа.

### 3.4 Модуль сетевой изоляции

Сетевая изоляция — наиболее критически важный с точки зрения реагирования на инциденты компонент системы. Её реализация потребовала детального изучения модели приоритетов правил Windows Firewall.

**Техническая реализация.** Стандартный подход — создание запрещающих правил для входящего и исходящего трафика — оказывается неработоспособным в данном контексте: явные Block-правила в Windows Firewall имеют приоритет над явными Allow-правилами на том же уровне. Следовательно, если добавить правила Block All, а затем Allow для IP аналитика, трафик к аналитику всё равно будет заблокирован (Cichonski et al., 2012).

Корректное решение основано на изменении дефолтного действия профиля брандмауэра через `Set-NetFirewallProfile`, а не на создании явных Block-правил. Allow-правила имеют приоритет над профильным умолчанием, что обеспечивает сохранение канала управления:

```powershell
New-NetFirewallRule -DisplayName 'BarysGuard_AllowMgmt_In'
    -Direction Inbound -Action Allow -RemoteAddress '<mgmt_ip>'
    -Enabled True -Profile Any

New-NetFirewallRule -DisplayName 'BarysGuard_AllowMgmt_Out'
    -Direction Outbound -Action Allow -RemoteAddress '<mgmt_ip>'
    -Enabled True -Profile Any

Set-NetFirewallProfile -Profile Domain,Private,Public
    -DefaultInboundAction Block -DefaultOutboundAction Block
```

При восстановлении сети оба Allow-правила удаляются, а профили брандмауэра возвращаются в состояние `NotConfigured`:

```powershell
Remove-NetFirewallRule -DisplayName 'BarysGuard_AllowMgmt_In' -ErrorAction SilentlyContinue
Remove-NetFirewallRule -DisplayName 'BarysGuard_AllowMgmt_Out' -ErrorAction SilentlyContinue
Set-NetFirewallProfile -Profile Domain,Private,Public
    -DefaultInboundAction NotConfigured -DefaultOutboundAction NotConfigured
```

Операции изоляции требуют привилегий администратора, поэтому агент перед выполнением скрипта проверяет статус через `ctypes.windll.shell32.IsUserAnAdmin()` и возвращает HTTP 500 с информативным сообщением об ошибке, если привилегий недостаточно. Это исключает молчаливые сбои, которые могут ввести аналитика в заблуждение относительно реального состояния изоляции.

### 3.5 Интеграция с внешними API

**VirusTotal API v3** используется в двух модулях: Hash Lookup и File Scan (хэширование удалённых файлов). Запросы формируются к эндпоинту `https://www.virustotal.com/api/v3/files/{hash}` с заголовком `x-apikey`. Ответ разбирается для извлечения поля `last_analysis_stats` с количеством детектов и `last_analysis_results` для детализации по движкам. Для учёта ограничений бесплатного API (4 запроса/мин) реализована настраиваемая задержка между запросами (по умолчанию 15 секунд).

**AbuseIPDB API v2** применяется для проверки репутации IP-адресов и доменов в модуле Network Intel. API-запрос к `/api/v2/check` возвращает `abuseConfidenceScore` (0-100%), информацию о стране, ISP и категориях злоупотреблений. IP-адреса с оценкой выше 80% помечаются как высокорисковые и отражаются в соответствующем счётчике Dashboard.

Оба API-клиента реализованы как фоновые `QThread`-воркеры, сигнализирующие о результате через Qt-сигналы. Ключи API хранятся в `config.json` и не передаются по сети ни при каких обстоятельствах.

### 3.6 Пользовательский интерфейс

Интерфейс построен на основе тёмной цветовой схемы GitHub-style (`#0d1117` фон, `#161b22` панели, `#58a6ff` акценты), что снижает нагрузку на зрение при длительной работе — фактор, принципиально важный для аналитиков SOC, проводящих за мониторингом многочасовые смены.

**Dashboard** содержит четыре динамические вкладки: Overview (живые счётчики хэш-проверок, YARA-хитов, IOC-запусков, высокорисковых IP и лента событий), Scans, Network, Hosts. Лента событий автоматически пополняется при каждой значимой находке из любой другой вкладки, создавая хронологическую картину активности аналитика.

**Многоязычность** реализована через словарь переводов `_TRANSLATIONS` в `core/i18n.py` с поддержкой трёх языков. Переключение языка происходит мгновенно через механизм Qt-сигналов: `lang_signal.changed` эмитируется при изменении настройки, и все компоненты, подключённые к нему, обновляют свои текстовые метки.

### 4.3 Анализ результатов и выводы

Результаты тестирования подтверждают работоспособность всех ключевых функций системы. Особо следует отметить несколько аспектов.

**Время отклика.** Сбор IoC с локального хоста (89 процессов, 34 активных соединения, 127 записей автозапуска) выполнился за 8,4 секунды, что значительно укладывается в нормативные 60 секунд. YARA-сканирование каталога из 500 файлов общим объёмом 1,2 ГБ заняло 47 секунд при 6 активных правилах, что соответствует ожидаемой производительности.

**Точность YARA-правил.** При сканировании 200 системных исполняемых файлов Windows было зафиксировано 0 ложноположительных срабатываний, что свидетельствует о высокой специфичности написанных правил. Это важный показатель: избыточные ложные срабатывания снижают доверие аналитика к системе и увеличивают время на обработку результатов (Pontiroli & Martinez, 2015).

**Надёжность сетевой изоляции.** В ходе тестирования корректность работы изоляции была проверена в трёх конфигурациях: одиночный хост в рабочей группе, хост в домене Active Directory, хост с несколькими сетевыми адаптерами. Во всех случаях изоляция применялась корректно, канал управления (соединение с IP аналитика) сохранялся, а восстановление сети проходило без ошибок.

**Удобство развёртывания агента.** Полный цикл подключения нового хоста через автоматизированное развёртывание (от нажатия кнопки «Deploy Agent» до получения рабочего Ping) занял в среднем 2 минуты 40 секунд — это существенно меньше, чем ручное подключение через классические SIEM-агенты (Wazuh Agent deployment, по документации производителя, занимает от 10 до 30 минут с последующей настройкой правил).

Таким образом, разработанная система полностью выполняет поставленные функциональные требования и демонстрирует приемлемые характеристики производительности в реальных условиях.

## ЗАКЛЮЧЕНИЕ

В ходе настоящей дипломной работы была разработана программная система BarysGuard, реализующая автоматизированный поиск индикаторов компрометации и оперативное реагирование на инциденты информационной безопасности в инфраструктуре Windows. Система прошла полный цикл разработки — от анализа требований и проектирования архитектуры до реализации и функционального тестирования.

В рамках работы были решены все поставленные задачи:

1. Проведён анализ теоретических основ работы с IoC, жизненного цикла реагирования на инциденты (на основе NIST SP 800-61 Rev. 2) и современных методов автоматизированного обнаружения угроз.
2. Выполнено сравнительное исследование существующих решений (Splunk ES, Microsoft Sentinel, Wazuh, TheHive+Cortex), определившее нишу и преимущества разрабатываемой системы.
3. Разработана многоуровневая архитектура приложения с разделением на UI, воркеры, ядро и удалённый агент.
4. Реализованы все запланированные функциональные модули: сбор IoC, YARA-сканирование (40+ правил), проверка хэшей, сетевая разведка, удалённое управление хостами, сетевая изоляция, форензика памяти, карантин, генерация отчётов.
5. Проведено функциональное тестирование из 20 тест-кейсов, все из которых завершились успешно.

Ключевым техническим достижением работы является корректная реализация механизма сетевой изоляции через изменение дефолтного действия профиля Windows Firewall в сочетании с Allow-правилами для IP аналитика — подход, обеспечивающий надёжное сдерживание угрозы при сохранении возможности удалённого управления скомпрометированным хостом.

Практическая значимость системы подтверждена тестированием в реальной двухузловой сети: время сбора первичного набора IoC не превышает 9 секунд, цикл развёртывания агента на новом хосте — менее 3 минут. Эти показатели сопоставимы с возможностями коммерческих систем при несопоставимо меньших требованиях к инфраструктуре и квалификации персонала.

В качестве направлений дальнейшего развития системы можно выделить: интеграцию с STIX/TAXII-фидами для автоматического обогащения базы IoC, добавление поддержки Linux-хостов в качестве целевой платформы для агента, реализацию модуля машинного обучения для поведенческого детектирования аномалий, а также разработку веб-интерфейса для доступа к системе из браузера.

Разработанный программный комплекс BarysGuard вносит практический вклад в решение актуальной задачи обеспечения доступности инструментов кибербезопасности для организаций, не располагающих ресурсами для внедрения полноценных корпоративных SIEM-систем.

**Б.2 Фрагмент YARA-движка (core/yara_engine.py)**

```python
def scan_path(self, path: str, rules: dict) -> list:
    compiled = yara.compile(source='\n'.join(rules.values()))
    results = []
    for root, _, files in os.walk(path):
        for fname in files:
            fpath = os.path.join(root, fname)
            try:
                matches = compiled.match(fpath, timeout=30)
                for m in matches:
                    severity = m.meta.get('severity', 'medium')
                    results.append({
                        'rule': m.rule,
                        'severity': severity,
                        'file': fpath
                    })
            except Exception:
                pass
    return results
```
